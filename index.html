HOW TIDAS WORKS

Apple's new APIs allows Touch ID to unlock iOS’s Secure Enclave so it can digitally sign session tokens for authentication. Authentication only happens when the user touches the iPhone’s button. Once Tidas is installed in your app:

The user opens your app and presents her fingerprint
Touch ID confirms the user’s fingerprint and converts it into a unique signature
Operating in the Secure Enclave, Tidas converts Touch ID’s signature into a new crypto key
The crypto key passes from your app to your website, and finally to us
When we authenticate the signature, we return an ‘OK’ to your website and app
The user enters your app in seconds

The public keys and signatures exposed to your application and to Tidas are useless without the private key stored safely and permanently inside the Secure Enclave. Because jailbreakers can’t access the Secure Enclave, they can’t steal the private keys.

In the first login, the user will authenticate with their email, or a legacy username and password. In subsequent logins, she’ll securely and uniquely re-identify with her fingerprint.

Absolute Privacy Guaranteed

If the Tidas database were breached tomorrow, the attackers would gain nothing.

That's because Tidas doesn’t store any data off of users’ devices. (Start a free trial to see our source code.) Users’ private keys stay in the Secure Enclave. Even if an attacker saw the entire Tidas database, he couldn't correlate groups of logins to user identities. Tidas just provides an 'authenticated/not authenticated' decision.

Tidas doesn’t collect fingerprints. That’s Touch ID’s job: it collects users' fingerprints for authentication and stores them in the Secure Enclave. That’s completely opaque to Tidas. By design, Tidas never knows who’s logging into your app, and you never have to touch users’ login credentials.

FAQ
What happens if users remove and reinstall the app?

The key is tied to your app’s provisioning profile and will remain available.

What if the user changes devices or loses her keys?

The Tidas API supports overwriting keys for users.

After validating your user's new device, simply re-enroll them in Tidas with the same unique identifier.

What happens if your service goes down?

We are deployed for high availability and encourage developers to avoid single points of failure.

We use AWS cloudawesomeness. If AWS goes down, so does everything else.

How does the first login work for users?

The device generates an enrollment request string.

The device returns the enrollment request to your server and passes it to the Tidas middleware component.

If the key data is valid, we store that user and confirm their identity.

a. If you provide an ID (e.g. their email), we’ll use that.

b. Otherwise, we’ll assign a random identifier.

What happens when our users delete their TouchID fingerprints?

As long as there’s one print, it will continue to work fine.

If they remove all their prints, Tidas will raise an error, letting the developer know that they need to ask the user for a new fingerprint.

Is Tidas supported on Android?

Contact us if you're interested in Android support.

How are you protecting my data?

Your data never leaves your system. The middleware strips out your data and passes us only the cryptographic hash and a signature for validation.

We collect the least data necessary, and make it valueless through the use of cryptography. If an attacker succeeds in hacking us, they get nothing. The credentials we store (public keys) can’t be reused like a password.

Beyond that, we used the simplest, most easily testable components possible to get the job done. Padrino+PostgreSQL on Heroku. All our code has tests and was reviewed by our Security R&D team prior to use.

How is this safer than Touch ID gated app access?

Attackers and Jailbreakers can’t grab the key from the Secure Enclave. Touch ID on its own could be patched out from the app’s binary code.

Where are the keys stored?

Your user’s private key is stored in her phone’s Secure Enclave. Her public key is exported and stored with the Tidas service. The private key will never leave the Secure Enclave. Ever.

What kind of cryptography does this use?

ECDSA secp256r1 (NIST P-256v1), the only curve that the Secure Enclave supports.

Why aren’t you using secp256k1?

Apple and NIST picked r1.

The only reason to use k1 is for bitcoin compatibility.

Could I use a different curve? For example, a djb curve like Curve25519?

No. Apple has chosen secp256r1 as the only curve supported by Secure Enclave.
